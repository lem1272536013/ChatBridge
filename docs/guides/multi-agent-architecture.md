# 多智能体架构：Chatflow 解耦方案

## 问题背景

Dify 的 Chatflow 应用本质上是一个单体流程——所有分支、判断、工具调用都必须在同一个画布内完成。随着业务复杂度增加，一个覆盖"咨询 → 意图识别 → 专项处理 → 结果反馈"全链路的 Chatflow 会迅速膨胀，分支节点嵌套、画布难以维护，不同业务模块之间也完全无法复用。Dify 官方目前没有提供 Chatflow 之间互相调用或解耦编排的机制。

## 解决思路：参照 A2A 协议，由前端承担智能体路由

Google 提出的 A2A（Agent-to-Agent）协议的核心理念是：每个智能体只负责单一职责，智能体之间通过标准化的消息协议交接任务和上下文，而不是把所有逻辑塞进一个单体智能体里。

ChatBridge 借鉴这一思想，将"智能体路由"的职责从后端 Chatflow 移到前端，实现 Chatflow 的彻底解耦：

- 每个 Chatflow 应用只做好一件事，保持精简。
- 前端负责感知当前智能体的输出信号，判断何时切换、切换到哪个智能体。
- 智能体之间的上下文交接通过前端传参完成，不依赖任何后端服务间调用。

## 实现机制

### 1. 后端输出信号（特殊标记）

Chatflow 在回复末尾嵌入约定格式，前端实时解析：

```
/// [{"api-key": "app-xxx", "input": "用户上一步的结论"}] ///
```
解析为"进入智能体"按钮，展示目标智能体名称，等待用户点击。

```
*** [{"task_status": "finish"}] ***
```
解析为任务完成按钮，提示用户返回主智能体或继续当前流程。

### 2. 前端路由切换（`selectAgent`）

> **安全提示**：多智能体跳转依赖将目标智能体的 `api-key` 写入前端 `sessionStorage`，这意味着 api-key 会暴露在浏览器端，任何能访问该页面的用户都可以通过开发者工具读取。建议：在 Dify 中为每个智能体设置独立的速率限制和用量上限；不要把高权限或计费敏感的 api-key 用于多智能体跳转；生产环境可考虑通过后端中转代理接口来隐藏真实 api-key。

用户点击"进入智能体"按钮时，前端执行以下步骤：

```
1. 冻结当前消息列表的快照（保留历史消息归属哪个智能体）
2. 在消息流中插入分隔线，标记"进入 XXX"
3. 将目标智能体的 api-key 写入 sessionStorage.api_key
4. 清空 conversation_id（开启全新会话）
5. 将上一个 Chatflow 指定的 input 字段注入新对话的初始消息
6. 调用 /v1/site 和 /v1/parameters 加载目标智能体配置
7. 发送初始消息，启动目标 Chatflow
```

### 3. 上下文跨智能体传递

上一个 Chatflow 可以在 `/// JSON ///` 标记中附加 `input` 字段，前端在跳转时将其作为新会话的第一条消息输入，目标 Chatflow 能直接拿到上游处理结果，不需要用户重新描述。

```json
[{
  "api-key": "app-下游智能体key",
  "input": "客户合同编号 HT-2024-0088，已完成意向确认"
}]
```

### 4. 任务完成后返回主智能体（`backAgent`）

专项 Chatflow 完成任务后，通过 `*** JSON ***` 标记告知前端，前端展示"任务已完成，返回主智能体"按钮。用户点击后，前端将 api-key 切换回 `MAIN_AGENT_API_KEY`，重置会话，用户回到入口智能体继续后续操作。

## 整体流转示意

```
主智能体（意图识别）
    │
    │  /// [{"api-key": "app-contract", "input": "合同审核需求"}] ///
    ▼
合同审核智能体（专项 Chatflow）
    │
    │  *** [{"task_status": "finish"}] ***
    ▼
返回主智能体 / 继续当前智能体
```

每个节点都是独立的 Dify Chatflow 应用，单独维护、单独迭代，互不干扰。前端消息列表用分隔线完整记录每一段对话归属哪个智能体，对话历史完整可读。

## 与传统单体 Chatflow 的对比

| | 传统单体 Chatflow | ChatBridge 多智能体方案 |
|---|---|---|
| 复杂度 | 随业务增长快速膨胀 | 每个 Chatflow 职责单一、保持精简 |
| 可维护性 | 分支嵌套深，难以调试 | 独立应用，独立测试 |
| 复用性 | 几乎不可复用 | 同一个专项智能体可被多个主智能体调用 |
| 上下文传递 | 在同一流程内天然共享 | 通过 `input` 字段跨应用传递关键信息 |
| 新增业务分支 | 修改现有 Chatflow | 新建独立应用，配置中增加一条映射 |
